## 1. Vault Identity

**Public whitewater kayaking community resource.** Safe for any LLM ‚Äî no personal or confidential information.

- **Maintainer:** Jose Luis Pino (hobby kayaker, not professional affiliation)
- **Audience:** Whitewater kayakers and paddlers using Garmin Instinct wearables
- **Mission:** Bridge the gap between river navigation and hands-free wrist-based GPS tools
- **Repository:** Public GitHub (`Kern-River-Boaters/gpx`)
- **Content scope:** GPS waypoint collections, Obsidian Leaflet maps, river guides, safety notes

**Key constraint:** This vault is openly published. Never reference personal vaults, work systems, or private data. All content must be appropriate for public redistribution.

## 3. File Types & Required Structure

### `type/river_guide` ‚Äî River guide with embedded map (most common)

Auto-generated by `refresh_gpx_maps.py`. User notes go above the auto-generated block.

**Required frontmatter:**

```yaml
---
tags:
  - type/river_guide
  - river/[river]             # e.g. river/kern/lower
  - feature/[feature]         # e.g. feature/rapid
  - difficulty/[class]        # e.g. difficulty/class_ii_iii  (omit if no rapids)
  - region/[region]           # e.g. region/california
  - season/[season]           # e.g. season/year_round
  - status/[status]           # e.g. status/verified
---
```

**Extended frontmatter** (for waypoint collections):
```yaml
river_section: "Lower Kern"
difficulty_range: "Class II-III"
waypoint_count: 42
last_verified: YYYY-MM-DD
```

**Required sections:**
- User notes / description (above auto-generated block) ‚Äî optional but encouraged
- `<!-- BEGIN AUTO-GENERATED MAP -->` block ‚Äî regenerated by script, do not hand-edit

## 4. Tag Hierarchy

**Schema location:** `_Meta/Schemas/obsidian_tag_refactor.json`

### `type/` ‚Äî Content category (REQUIRED on every file)
- `type/river_guide` ‚Äî Markdown files with embedded maps and river notes
- `type/gpx` ‚Äî Raw GPX source files
- `type/data` ‚Äî GeoJSON data files
- `type/map` ‚Äî Legacy map-only files (prefer `type/river_guide` for new content)
- `type/tutorial` ‚Äî Setup guides and how-tos
- `type/script` ‚Äî Python scripts
- `type/documentation` ‚Äî README, CONTRIBUTING, SOP files
- `type/meta` ‚Äî Vault governance files

### `river/` ‚Äî River system
- `river/kern` ‚Äî Kern River (general)
- `river/kern/upper` ‚Äî Upper Kern (Forks to Johnsondale)
- `river/kern/lower` ‚Äî Lower Kern (Kernville to Lake Isabella)
- `river/kern/north_fork` ‚Äî North Fork of Kern
- `river/san_joaquin` ‚Äî San Joaquin River
- `river/rogue` ‚Äî Rogue River (Oregon)

**Rule:** Use the most specific sub-hierarchy available.

### `difficulty/` ‚Äî Whitewater classification (use on rapids content)
- `difficulty/class_i` ‚Äî Beginner (moving water, easy)
- `difficulty/class_ii` ‚Äî Novice
- `difficulty/class_iii` ‚Äî Intermediate
- `difficulty/class_iv` ‚Äî Advanced
- `difficulty/class_v` ‚Äî Expert (potentially life-threatening)
- Range tags: `difficulty/class_ii_iii`, `difficulty/class_iii_iv`, `difficulty/class_iv_v`

**Rule:** For content covering a range of rapids, use the combined range tag. Omit on non-rapid content (campgrounds, parking).

### `feature/` ‚Äî What the waypoints represent
- `feature/rapid` ‚Äî Rapids and drops
- `feature/campground` ‚Äî Camping areas
- `feature/access` ‚Äî Put-ins and take-outs
- `feature/parking` ‚Äî Parking areas and trailheads
- `feature/hazard` ‚Äî Portages, sieves, keeper holes, dangerous features
- `feature/poi` ‚Äî Points of interest (waterfalls, hot springs, landmarks)
- `feature/milestone` ‚Äî River miles, gauges, confluences

**Rule:** A file may have multiple `feature/` tags if it covers mixed waypoint types.

### `season/` ‚Äî When the river runs
- `season/spring_runoff` ‚Äî Spring snowmelt (high flows, often dangerous)
- `season/summer` ‚Äî Summer low-flow conditions
- `season/fall` ‚Äî Fall flows
- `season/winter` ‚Äî Winter flows
- `season/year_round` ‚Äî Runnable year-round

### `region/` ‚Äî Geographic area
- `region/california` ‚Äî California
- `region/sierra_nevada` ‚Äî Sierra Nevada mountains
- `region/southern_california` ‚Äî Southern California
- `region/oregon` ‚Äî Oregon

**Rule:** Apply multiple region tags for geographic nesting (Kern ‚Üí `region/california` + `region/sierra_nevada`).

### `flow/` ‚Äî Flow-specific content (optional)
- `flow/low`, `flow/medium`, `flow/high`, `flow/flood`

**Rule:** Only tag when content is specifically about conditions at a particular flow level.

### `status/` ‚Äî Data accuracy and currency
- `status/verified` ‚Äî Field-verified, GPS coordinates confirmed accurate
- `status/needs_verification` ‚Äî Untested or unconfirmed in the field
- `status/outdated` ‚Äî Known to need updates (conditions changed, trail rerouted, etc.)
- `status/draft` ‚Äî Work in progress, not ready for use on the river

**Rule:** Default new/untested data to `status/needs_verification`. Always set a status.

## 5. Content Standards

**Safety accuracy is the highest priority in this vault.** Outdated hazard information can injure or kill paddlers.

- **Difficulty ratings** must reflect current conditions. Ratings can change with flow level, fallen trees, or changed channels. Always note the flow conditions a rating applies to.
- **Hazard waypoints** (`feature/hazard`) ‚Äî describe portage route, why it is dangerous, and current scouting information.
- **GPS accuracy caveat:** Kern and Rogue canyons have poor satellite reception. All guides must preserve the warning that GPS accuracy degrades in deep canyons.
- **River conditions change seasonally.** Files covering specific flow windows should use `flow/` tags and note the flow range they were surveyed at.
- **Date content.** Waypoint collections should have `last_verified: YYYY-MM-DD` in frontmatter. Unverified data within the last season should be noted; data older than 3 years should be tagged `status/needs_verification` or `status/outdated`.

**Voice and format:**
- Direct and informational ‚Äî written for paddlers who need to act on information quickly
- Safety warnings use plain language, not jargon
- Metric and imperial measurements are both acceptable; be consistent within a file
- No personal anecdotes or subjective opinions without clearly labeling them as such

## 6. Tagging Heuristics (AI Inference)

When inferring tags from file names and content:

- Files in `Kern/` ‚Üí `river/kern`. "Lower Kern" ‚Üí `river/kern/lower`, "North Fork" ‚Üí `river/kern/north_fork`
- Files in `San Joaquin/` ‚Üí `river/san_joaquin` + `region/california` + `region/sierra_nevada`
- Files in `Rouge River/` ‚Üí `river/rogue` + `region/oregon`
- "Rapids" in filename ‚Üí `feature/rapid`; "Campground" ‚Üí `feature/campground`; "Access" ‚Üí `feature/access`; "Parking" ‚Üí `feature/parking`
- Class IV-V rapids content ‚Üí always add `difficulty/class_iv_v`; do not omit on technical whitewater
- GPX source files ‚Üí `type/gpx`; `.json` GeoJSON data ‚Üí `type/data`; `.md` with leaflet block ‚Üí `type/river_guide`
- Default to `season/year_round` for Kern mainstem; add flow-specific season tags if mentioned in content
- Default to `status/verified` only for field-confirmed data with `last_verified` date set

## 7. Audit Rules

### Flag as FAIL (high priority ‚Äî safety)
- Difficulty rating missing on any file with `feature/rapid` ‚Äî infer from content or flag `status/needs_verification`
- `status/outdated` or missing `last_verified` on rapid/hazard waypoint files ‚Äî safety risk
- Hazard waypoint (`feature/hazard`) with no portage or safety description in the notes section
- Any content suggesting GPS is 100% reliable in canyons ‚Äî must preserve canyon accuracy disclaimer

### Flag as FAIL (data quality)
- Missing `type/` tag on any file
- Missing `status/` tag on any file
- Missing `river/` tag on a river guide file
- Flat tags like `#kern`, `#rapid`, `#class-iv` ‚Äî must be hierarchical
- Nested frontmatter objects (breaks Dataview)

### Flag as WARNING (review needed)
- `status/needs_verification` on content that has been in the vault more than one paddling season without update
- `last_verified` date older than 3 years ‚Äî suggest field re-verification or downgrade to `status/outdated`
- `feature/rapid` file with no difficulty tag and no note explaining why
- River sections with no `season/` tag

### Do NOT flag
- Auto-generated content inside `<!-- BEGIN AUTO-GENERATED MAP -->` / `<!-- END AUTO-GENERATED MAP -->` blocks ‚Äî script-managed
- `.json` GeoJSON files with no frontmatter ‚Äî data files, not notes
- `.gpx` binary/XML files ‚Äî not markdown
- `_Meta/Tests/` files ‚Äî plugin behavior test cases, non-navigation content
- Missing `feature/` on access, parking, campground-only collections when covered by a more specific `feature/` tag

## Model Context Protocol (MCP) Integration

This vault runs a dedicated Model Context Protocol (MCP) server locally inside Obsidian via the `obsidian-mcp-pro-plugin` plugin.

* **Assigned Port:** `3336`
* **Server Name:** `obsidian-krb`
* **Settings File:** `.obsidian/plugins/obsidian-mcp-pro-plugin/data.json`
* **Connection Protocol:** Read the local settings file to retrieve the auto-generated `bearerToken` and the configured port. Construct the authorization header (`Bearer <token>`) and connect to `http://127.0.0.1:3336/mcp`.
* **Server Not Running Protocol:** If the connection to the MCP server fails, and accessing it is useful/necessary for your current task, you MUST explicitly prompt the user to open Obsidian and start the MCP Pro plugin server for this vault (Settings ‚Üí MCP Pro ‚Üí Start).
